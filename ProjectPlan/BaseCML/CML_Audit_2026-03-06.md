# CML Audit - 2026-03-06

## Scope

This audit focused on whether CML actually retrieves and applies latent constraints under cue-trigger semantic disconnect, as claimed in the Cognitive Constraint Layer section of [README.md](README.md) lines 128-141 and the LoCoMo-Plus evaluation notes in [evaluation/README.md](evaluation/README.md) lines 140-147.

Evidence came from repository code, tests, and targeted local test runs only. When I could not run an end-to-end scenario, I label the conclusion as an inference from static inspection.

## Executive Summary

The repo has real constraint-aware machinery, but the best parts are not consistently on the hot path used by the product.

1. The default non-LLM write path under-materializes semantic memory. `SemchunkChunker` emits only `STATEMENT` chunks (`src/memory/working/chunker.py:40-86`), tests document that behavior (`tests/integration/test_short_term_memory_flow.py:62-67`), and `STATEMENT` maps to episodic memory (`src/memory/hippocampal/write_gate.py:319-330`). The local unified extractor can return `facts` and `memory_type` (`src/extraction/local_unified_extractor.py:57-79`, `src/extraction/local_unified_extractor.py:82-93`, `src/extraction/local_unified_extractor.py:121-126`), but the write path mostly ignores those outputs and only uses local importance/context/confidence/decay (`src/memory/hippocampal/store.py:136-145`, `src/memory/hippocampal/store.py:242-247`, `src/memory/hippocampal/store.py:450-587`).
2. The retrieval stack can surface semantically distant constraints, but often by broad "fetch all active constraint categories" fallback rather than by a strong associative bridge. Planner comments say "no general fallback" (`src/retrieval/planner.py:180`), yet `constraint_categories=None` causes the retriever to query all five cognitive categories (`src/retrieval/retriever.py:490-497`).
3. `/memory/read` can use the repo's good constraint-first formatter (`src/api/routes.py:188-194`, `src/retrieval/packet_builder.py:161-296`), but `/memory/turn` bypasses it and falls back to `MemoryPacket.to_context_string(...)`, which puts constraints last (`src/memory/seamless_provider.py:150-188`, `src/core/schemas.py:190-205`).
4. Constraint supersession is too coarse. `constraint_fact_key(...)` only keys by `subject:type:scope_hash` (`src/extraction/constraint_extractor.py:179-187`), so unrelated constraints in the same coarse scope can overwrite each other when semantic facts supersede on key (`src/memory/neocortical/fact_store.py:421-446`).
5. Constraint subtype, provenance, and validity metadata are weakened as semantic facts move through retrieval and reranking. `_fact_to_record(...)` does not carry category/evidence/validity into `record.metadata` (`src/retrieval/retriever.py:722-748`), while reranker stability logic expects subtype metadata (`src/retrieval/reranker.py:83-108`).

Bottom line: CML can pass a narrow semantic-disconnect smoke test, but it remains vulnerable to constraint dilution, wrong-type retrieval, stale-overwrite mistakes, and path-dependent behavior between `/memory/read` and `/memory/turn`.

## Research Alignment Critique

### Multi-store model

README maps Atkinson-Shiffrin and Miller to `SensoryBuffer`, `WorkingMemoryManager`, and `SemchunkChunker` ([README.md](README.md) lines 67-81). The pipeline exists, but the semantic handoff is weak because chunk typing collapses to `STATEMENT` in the default path (`src/memory/working/chunker.py:40-86`).

### Complementary learning systems

README claims fast episodic plus slow semantic stores ([README.md](README.md) lines 83-96). The stores exist, but the default local write path does not fully wire local semantic extraction into memory typing or semantic-fact creation (`src/memory/orchestrator.py:168-192`, `src/memory/orchestrator.py:285-293`, `src/memory/hippocampal/store.py:136-145`, `src/memory/hippocampal/store.py:450-587`).

### HippoRAG

README says CML borrows graph retrieval plus personalized PageRank from HippoRAG ([README.md](README.md) lines 118-140, 549-553). The graph step exists, but it is only added when query entities are present (`src/retrieval/planner.py:208-216`) and PPR is seeded from explicit entities only (`src/storage/neo4j.py:259-340`). Inference: current semantic-disconnect success is driven more by broad constraint lookup than by graph-mediated associative recall.

### Mem0 / belief revision

README cites Mem0 for add/update/delete and belief revision ([README.md](README.md) lines 124-126). CML has those mechanics, but the update identity for constraints is too coarse because `constraint_fact_key(...)` only uses type and scope hash (`src/extraction/constraint_extractor.py:179-187`).

### LoCoMo-Plus

README and `evaluation/README.md` emphasize type-specific cognitive constraints ([README.md](README.md) lines 126-141; [evaluation/README.md](evaluation/README.md) lines 140-147). The repo has a good constraint-first packet builder (`src/retrieval/packet_builder.py:161-296`, `tests/unit/test_constraint_layer.py:897-1027`), but the main conversational path bypasses it (`src/memory/seamless_provider.py:150-188`).

## Failure-Mode Analysis

### Semantic disconnect failure

Current cause:

- planner only adds `GRAPH` when query entities exist: `src/retrieval/planner.py:208-216`
- graph PPR is seeded from explicit entities only: `src/storage/neo4j.py:259-340`
- broad constraint fallback activates when categories are unspecified: `src/retrieval/retriever.py:490-497`
- current integration coverage only proves one stored policy can appear somewhere in the packet: `tests/integration/test_constraint_semantic_disconnect.py:55-94`

Fix:

- store first-class constraint objects with activation/domain metadata
- route decision/recommendation queries to typed constraint retrieval even without lexical overlap
- add associative expansion from top constraints, same subject, same domain, and graph-linked entities

Test:

- ingest one relevant policy, several irrelevant policies, and many semantically similar episodes
- query "Recommend a seafood restaurant"
- assert the shellfish policy is top-ranked and appears in the final answer context

### Constraint dilution

Current cause:

- `/memory/turn` filters `packet.all_memories` and then uses `MemoryPacket.to_context_string(...)`: `src/memory/seamless_provider.py:139-188`
- `MemoryPacket.to_context_string(...)` orders sections as facts, preferences, recent events, procedures, constraints: `src/core/schemas.py:190-205`

Fix:

- make `/memory/turn` format through `MemoryPacketBuilder.to_llm_context(...)`
- keep a hard constraint budget in every conversational path

Test:

- one must-follow policy plus many irrelevant episodes should still yield a constraint-first `memory_context`

### Wrong constraint type

Current cause:

- `_fact_to_record(...)` does not preserve category/evidence/validity metadata: `src/retrieval/retriever.py:722-748`
- reranker stability logic depends on `metadata["constraints"]`: `src/retrieval/reranker.py:83-108`
- summarizer fallback maps any constraint-sourced gist to `"policy"`: `src/consolidation/summarizer.py:314-322`

Fix:

- preserve subtype, provenance, validity, and lineage on semantic-fact retrieval records
- keep consolidation type-preserving for constraints

Test:

- ingest value, goal, state, and policy constraints; assert type-specific queries rank the correct subtype first

### Temporal / recency bias mistakes

Current cause:

- reranker reduces recency for constraints only when subtype metadata is available: `src/retrieval/reranker.py:73-109`
- category fetches use `LIMIT` without `ORDER BY`: `src/memory/neocortical/fact_store.py:82-147`
- broad constraint fallback mixes stable and volatile types in the initial pool: `src/retrieval/retriever.py:490-497`

Fix:

- order constraints by activation fit, type stability, confidence, and recency
- preserve subtype metadata end-to-end

Test:

- old but active value constraint plus recent transient state constraint; a value-sensitive decision query should keep the value above the transient state

### Stale constraint application

Current cause:

- `constraint_fact_key(...)` only uses `subject:type:scope_hash`: `src/extraction/constraint_extractor.py:179-187`
- unscoped constraints collapse to `"general"`: `src/extraction/constraint_extractor.py:185-187`
- fact updates supersede on key: `src/memory/neocortical/fact_store.py:421-446`

Fix:

- redesign keys around canonical constraint identity: subject + type + normalized predicate + scope + activation
- keep coexistence and supersession separate

Test:

- two active finance goals should coexist; only a later correction to one goal should supersede its own lineage

### Inconsistent consolidation

Current cause:

- summarizer fallback collapses constraints to `"policy"`: `src/consolidation/summarizer.py:314-322`
- consolidation worker stores constraint facts under the same coarse keying model: `src/consolidation/worker.py:134-177`
- mixed-topic fallback uses a single anchor and `"policy"`/`"summary"` gist types: `src/consolidation/worker.py:367-380`

Fix:

- add typed constraint gists
- reject or split mixed-topic constraint clusters

Test:

- consolidate a mixed goal/value cluster and assert either two typed gists are produced or the cluster is rejected

## Prioritized Roadmap

Quick wins:

1. Route `/memory/turn` through `MemoryPacketBuilder.to_llm_context(...)`.
2. Preserve reranked order separately from categorical buckets.
3. Fix batched evidence/provenance IDs.
4. Add deterministic ordering to semantic-fact category fetches.
5. Preserve subtype and provenance metadata in `_fact_to_record(...)`.
6. Remove broad constraint fallback from generic questions.

Medium:

1. Redesign `ConstraintObject` identity and supersession rules.
2. Wire local unified extractor outputs into memory typing and write-time fact creation.
3. Add associative expansion after first-stage retrieval.
4. Add trigram or FTS indexes for semantic facts.
5. Move graph sync off the write hot path or batch it properly.

Larger refactors:

1. Add a first-class active-constraint store/view model.
2. Make consolidation and reconsolidation operate on typed constraint objects instead of raw text only.
3. Add benchmark suites for semantic-disconnect precision, correct constraint type, supersession correctness, and latency.

## Benchmarks, Risks, and Non-Goals

Functional benchmarks to add:

- semantic-disconnect precision@k with distractors
- correct constraint subtype @1
- stale-vs-current constraint selection
- `/memory/read` vs `/memory/turn` context-equivalence checks

Performance benchmarks to add:

- write latency with and without graph sync
- retrieval latency by source: vector, facts, constraints, graph
- semantic-fact search latency before and after indexing

Risks:

- better constraint recall can increase latency unless candidate caps stay tight
- key redesign will require migration and compatibility handling
- stronger overwrite semantics can change behavior for tenants with ambiguous existing keys

Non-goals in the first patch wave:

- no redesign of unrelated dashboard UX
- no backend replacement for pgvector or Neo4j
- no broad rewrite of forgetting unless it directly affects active constraints
- no mandatory external LLM dependency for core retrieval correctness

## Bottom Line

The repo already contains most of the ingredients needed for strong latent-constraint retrieval. The main problems are integration quality and default-path behavior:

- the best formatter is not on the conversational hot path
- the default write path under-materializes semantic memory
- constraint identity is too coarse
- graph retrieval is too entity-seeded to carry the associative story by itself

If the goal is measurable improvement in constraint-consistent behavior, PR #1 through PR #4 should come before larger graph or model upgrades.
